## Summary

VATE v0.2.0 narrows the next draft around verifier-side admission for `AL2` external digital actions.

It keeps VATE adjacent to A2A, MCP, OAuth, VC, DID, OID4VP, Web Bot Auth, AP2, x402, ACP, and delegated payment-token systems.
Those systems remain evidence or transport layers.
VATE defines how a relying party evaluates evidence before execution and records the decision.

## Changed

- README now points readers to the v0.2 profile.
- Roadmap now tracks `v0.2-al2-verifier-admission-profile`.
- Non-goals now explicitly exclude A2A core changes, MCP authorization replacement, new signature schemes, global trust scoring, and payment protocol ownership.
- Transport binding notes now recommend digest-bound reference-only A2A metadata for v0.2.
- Verifier validation flow now describes dependency-aware stages rather than a rigid one-dimensional order.
- Conformance notes now distinguish repository fixture integrity checks
  (`run`) from external SUT comparison (`compare`) and local report-bundle
  digest-chain verification (`verify-bundle`).
- Payment-related evidence types were generalized in core schemas, with concrete protocols left as informative hints or examples.
- The v0.2 JOSE boundary is explicit: current fixtures are byte-level detached
  proof fixtures, not production ECDSA / PKI verification.
- A2A signed Agent Card evidence is treated as adjacent evidence and is bound
  to admission receipt evidence by digest; it does not authorize execution by
  itself.
- A2A metadata artifact reference `uri` fields now require absolute URI-shaped
  strings in the metadata schema. This remains shape validation only and does
  not imply remote dereference or trust.

## What Implementers Can Test

The draft conformance corpus gives named expected outcomes for 63 AL2 v0.2
cases. Implementers can run their verifier against `corpus.json`, emit a SUT
result file matching `schemas/sut-result.schema.json`, and compare it with the
path described in:

- `docs/conformance/external-sut-quickstart.md`

Minimal comparison command:

```bash
python3 scripts/vate_conformance.py compare \
  --corpus-root conformance/al2-vate-v0.2 \
  --sut-results examples/conformance/sut-results-pass.example.json \
  --report /tmp/vate-sut-compare-report.json \
  --implementation-report /tmp/vate-sut-implementation-report.json
```

The comparison path checks expected decisions, execution gates, reason codes,
case-specific checks, required artifact references, digest descriptors, AL2
context bindings, and JOSE proof-package references for cases that require
them.

Passing this comparison means one SUT result matched one corpus snapshot. It
does not grant a compatibility badge or production approval.

## Verification

The full pre-release gate is defined in:

- `docs/release-gate-v0.2.0.md`

Whitespace and diff check:

```bash
git diff --check
```

Dependency-free verification:

```bash
python3 scripts/check_repo.py
```

This is a fast repository smoke and synchronization check, not full JSON Schema
validation. The release gate uses `scripts/check_repo_strict.py` when
`jsonschema` is available.

Repository fixture check:

```bash
python3 scripts/vate_conformance.py run \
  --corpus-root conformance/al2-vate-v0.2 \
  --report /tmp/vate-conformance-report.json
```

External SUT comparison:

```bash
python3 scripts/vate_conformance.py compare \
  --corpus-root conformance/al2-vate-v0.2 \
  --sut-results examples/conformance/sut-results-pass.example.json \
  --report /tmp/vate-sut-compare-report.json \
  --implementation-report /tmp/vate-sut-implementation-report.json
```

Local report-bundle digest-chain verification:

```bash
python3 scripts/vate_conformance.py verify-bundle \
  --corpus-root conformance/al2-vate-v0.2 \
  --sut-results examples/conformance/sut-results-pass.example.json \
  --conformance-report /tmp/vate-sut-compare-report.json \
  --implementation-report /tmp/vate-sut-implementation-report.json \
  --report /tmp/vate-report-bundle-verification.json
```

TypeScript helper verification:

```bash
npm ci
npm run ts:check
npm run ts:test
npm audit --audit-level=moderate
```

Optional strict schema validation:

```bash
python3 -m venv ../verifiable-agent-trust-envelope-draft-venv
. ../verifiable-agent-trust-envelope-draft-venv/bin/activate
python3 -m pip install -r requirements-dev.txt
python3 scripts/check_repo_strict.py
```

The reference core's local decimal-size bounds are defensive guardrails only.
The v0.2 corpus checks aggregate `max_amount` semantics, currency mismatch, and
finite non-negative amount handling, but it does not require every
implementation to adopt the same local text-length, integer-digit, or
fractional-digit limits.
