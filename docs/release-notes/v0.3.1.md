# VATE v0.3.1 Release Notes Draft
## Credibility and reviewability patch

This file describes the intended `v0.3.1` discussion-draft patch on the current
main branch.

Post-archive citation note: the archived Zenodo record for `v0.3.1` has DOI
`10.5281/zenodo.20173995`. The current repository citation metadata is
`CITATION.cff`.

`v0.3.1` does not broaden VATE beyond the archived `v0.3.0`
`VATE-AL2-Verifier-Admission-v0.3` discussion-draft boundary. It remains a
review aid for verifier-side admission, attenuation, receipt, external SUT
comparison, and report-bundle integrity semantics.

## Summary

`v0.3.1` focuses on implementer credibility and reviewer navigation:

- canonical AL2 attenuation effective constraint names;
- small fail-closed attenuation fixtures for legacy emitted aliases;
- a receipt audit walkthrough for following digest-bound admission,
  post-execution, policy snapshot, and report-bundle references.

The main machine-readable corpus is now a 66-case AL2 v0.3 draft conformance
corpus. The added cases keep the existing AL2 verifier-admission boundary
and only reject non-canonical emitted attenuation forms.

## Changed

- `docs/attenuation-semantics.md` now defines canonical emitted
  `attenuation.effective_constraints` names for AL2 receipts.
- Admission receipt and attenuation-effect schemas now type known canonical
  effective constraint fields while preserving explicit extension room.
- Admission receipt schema now requires non-empty structured
  `attenuation.changes` entries for the current AL2 attenuation boundary.
- `schemas/attenuation-effect.schema.json` now identifies the normalized
  canonical effect shape as `app-effect-0.2` and keeps the status effect
  object constraints-only; status-layer aliases remain input signals that must
  be normalized before receipt emission.
- The reference runner fails closed when emitted attenuation constraints use
  legacy aliases such as `max_amount_usd`, bare `resource`, or string-valued
  `approval`.
- README and A2A review package entry points link the receipt audit walkthrough.

## What Implementers Can Test

The draft conformance corpus gives named expected outcomes for 66 AL2 v0.3
cases. Implementers can run their verifier against `corpus.json`, emit a SUT
result file matching `schemas/sut-result.schema.json`, and compare it with the
reference runner.

Minimal comparison command:

```bash
python3 scripts/vate_conformance.py compare \
  --corpus-root conformance/al2-vate-v0.3 \
  --sut-results examples/conformance/sut-results-pass.example.json \
  --report /tmp/vate-v0.3.1-compare-report.json \
  --implementation-report /tmp/vate-v0.3.1-implementation-report.json
```

Passing this comparison means one SUT result matched one corpus snapshot. It
does not grant a compatibility badge, certification, endorsement, or production
approval.
