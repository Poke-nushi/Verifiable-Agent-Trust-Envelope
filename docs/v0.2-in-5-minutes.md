# VATE v0.2 In 5 Minutes
## AL2 verifier admission profile draft

## Short Take

VATE v0.2 narrows the draft to one boundary:

> Before an external agent performs a risky digital action, what should the relying party verify, and what receipt should exist afterward?

The v0.2 answer is a verifier-side admission profile.
It does not ask A2A, MCP, OAuth, VC, DID, OID4VP, Web Bot Auth, AP2, x402, ACP, or payment-token systems to change their core protocols.

Instead, those systems can provide evidence.
VATE defines how a verifier evaluates that evidence and records the result.

## What v0.2 Adds

- `VATE AL2 Verifier Admission Profile v0.2`
- reference-only A2A metadata binding
- admission receipt schema
- post-execution receipt schema
- admission request schema
- artifact and evidence reference schemas
- examples for `allow`, `attenuate`, and `deny`
- a mini conformance corpus for the minimum verifier outcomes

## Why This Is More Than A Comment

The v0.2 draft is meant to be reviewable by implementers.
It includes:

- profile text
- A2A metadata shape
- receipt model
- JSON Schemas
- concrete examples
- conformance cases with expected outcomes

That does not make it production-ready.
It does make the proposal testable enough for technical review.

## Main Design Rule

A2A carries task flow.
VATE carries verifier decision semantics.

In an A2A interaction, v0.2 recommends putting only references in A2A metadata:

- artifact URI
- media type
- digest
- issuer
- issued time
- expiration time
- phase

The full receipt, attenuation, permit, and evidence bodies stay outside A2A core.

## Decision Outcomes

### `allow`

The verifier admits the requested action as-is.

### `attenuate`

The verifier admits a narrower version of the request.

Examples:

- lower a payment limit
- narrow a tool allowlist
- shorten an execution window
- restrict a target resource

The narrowing must be machine-readable.
It should record the original request hash, the effective request hash, explicit changes, reason codes, and effective constraints.

### `deny`

The verifier rejects the request before execution.

Examples:

- expired permit
- audience mismatch
- revoked status
- stale runtime proof
- digest mismatch

## Receipt Split

v0.2 separates two receipt phases.

`admission_receipt`:

- issued before execution
- normally verifier-signed or verifier-issued
- records `allow`, `attenuate`, or `deny`
- records policy id, policy version, reason codes, and evidence checks

`post_execution_receipt`:

- issued after execution, cancellation, or failure
- links back to the admission receipt
- records effective request hash, output digest, side effects, and policy violations

## What v0.2 Does Not Define

v0.2 does not define:

- A2A core state-machine changes
- a new identity registry
- a new signature scheme
- a payment protocol
- a global trust score
- an MCP authorization replacement
- a runtime disclosure manifest format

The most useful feedback is:

- is attenuation specific enough for implementation and audit
- are the schema names too broad or too narrow
- should this remain an adjacent VATE profile or move toward an A2A community extension
